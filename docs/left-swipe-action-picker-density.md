# 左滑动作面板：界面利用率提升设计方案

> 目标界面：`Quickstart/UI/AiActionPickerPopup.cs`  
> 触发：右键左滑 → 动作选择面板（Prompt / 工具 / 后续可含 Skill）

---

## 1. 现状诊断

### 1.1 当前结构

| 维度 | 现状 |
|------|------|
| 尺寸 | 固定逻辑尺寸 **520×320** |
| 布局 | 左右 **50% / 50%** 双栏（Prompt / 工具） |
| 行高 | 普通项 **48px**，Skill **58px** |
| 行间距 | 项底部 **6px** |
| 内边距 | root padding **10px**，header 下 margin **8px** |
| 内容策略 | 条目少时仍占满固定高度，空白暴露明显 |
| 信息层级 | 标题 + 纯文字行，缺少图标/最近/数量 |

### 1.2 利用率偏低的根因

1. **高度不随内容变化**  
   3～5 个动作时，每栏底部约 1/3 为空白，视觉“空”且手势瞄准区域分散。

2. **行高偏松**  
   手势松手命中场景下，48px 行高可接受，但双栏固定宽 + 大行高导致**同屏可选项偏少**；动作一多又要滚动，体验两头不讨好。

3. **双栏等分不随两侧数量调节**  
   Prompt 很多、工具很少（或相反）时，一侧拥挤滚动、一侧大片空白。

4. **信息密度低**  
   仅名称文本，无类型色点、无最近使用、无快捷提示；“好看”靠留白，却牺牲了扫视效率。

5. **与后续 AI 面板割裂**  
   选中 Prompt 后会打开 `AiPopup`（620×640，输入/结果上下分栏）。左滑第一屏若太空，整体链路会显得“点一下才进正题”。

---

## 2. 设计原则

1. **内容驱动尺寸**：少内容时面板收小，多内容时在上限内扩展，再滚动。  
2. **手势优先密度**：行高以“可稳准命中”为下限，不为装饰再加高。  
3. **一眼可扫**：优先暴露高频动作，其次完整分类列表。  
4. **美观 = 节奏，不是大留白**：统一圆角、轻分隔、类型色点；避免空洞灰区。  
5. **渐进增强**：先做低风险密度优化，再加“最近使用 / 非对称列宽”等进阶能力。

---

## 3. 方案总览

| 方案 | 定位 | 改动成本 | 利用率 | 美观风险 |
|------|------|----------|--------|----------|
| **A. 紧凑自适应双栏（推荐首期）** | 改尺寸/行高/自适应高度 | 低 | ★★★★ | 低 |
| **B. 最近 + 全量分区** | A + 顶部 chip | 中 | ★★★★★ | 中（需控制层级） |
| **C. 非对称/流式栅格** | 侧栏数量差大时更优 | 中高 | ★★★★★ | 中 |

**落地状态（2026-07-16）：A + B 已在 Windows `AiActionPickerPopup` 实现。**

---

## 4. 方案 A：紧凑自适应双栏（首期推荐）

### 4.1 尺寸与密度参数

| 参数 | 现状 | 建议 |
|------|------|------|
| 默认逻辑宽 | 520 | **400～440**（内容少时更轻） |
| 最大逻辑宽 | 520 | **480** |
| 最小逻辑高 | 固定 320 | **约 160**（标题 + 2 行） |
| 最大逻辑高 | 320 | **360～400**（超出滚动） |
| 项行高 | 48 / 58 | **36** 普通 / **44** Skill（两行 meta 时） |
| 项间距 | 6 | **4** |
| 内边距 | 10 | **8～10**（外框可略紧，项内 padding 保留 10～12） |
| 标题区 | 28 + margin 8 | **24 + margin 6** |

**经验公式（逻辑像素）：**

```
contentHeight = headerH + max(leftCount, rightCount) * (rowH + gap) - gap + bottomPad
panelHeight   = clamp(contentHeight, minH, maxH)
panelWidth    = both columns have ≥1 items ? preferredW : compactW
```

### 4.2 布局结构

```
┌──────────────────────────────────────┐
│  动作选择                      [设置] │  ← 单行顶栏（可拖动）
├──────────────────┬───────────────────┤
│ Prompt · n       │ 工具 · m          │  ← 小号分组标题 + 计数
│ ┌──────────────┐ │ ┌───────────────┐ │
│ │ ● 翻译       │ │ │ ● 粘贴纯文本  │ │  ← 行高 36，左色点
│ │ ● 总结       │ │ │ ● Everything  │ │
│ │ ● 润色       │ │ │ ● 网页搜索    │ │
│ └──────────────┘ │ └───────────────┘ │
└──────────────────┴───────────────────┘
        高度 = f(两侧最大行数)
```

### 4.3 视觉规范（保证美观）

- **背景**：保持 `#FAFAFA` 面板 + 白卡片行，或行仅 hover/highlight 时填浅底。  
- **高亮**：沿用手势路径高亮；描边/底色略加强（现有浅蓝可保留）。  
- **类型色点**（8px 圆）：  
  - Prompt：蓝 `#378ADD`  
  - 工具：绿 `#639922`  
  - Skill：紫 `#7F77DD`  
- **分隔**：双栏中间 **0.5px / #E6E4DC**，不要粗竖线。  
- **圆角**：项 8px，面板 12～14px（与现有 `FormStyler` 一致）。  
- **字体**：标题 10～11 Bold；项名 9.5～10 Regular；分组计数 8.5～9 Secondary。  
- **空状态**：某一栏无项时，用单行灰字“暂无”，**不要**用等高空白撑开；可选将该栏收窄或临时改单栏。

### 4.4 交互不变项（兼容现有逻辑）

- 手势阶段：`NOACTIVATE`，不抢前台。  
- 松手命中：`TryReleaseAtScreenPoint`。  
- 未命中：`EnterInteractiveMode`，可拖动 / ESC / 失焦关闭。  
- 设置按钮：仍打开设置后隐藏面板。

### 4.5 预期收益

- 同样 6 个动作：从“半屏空白”变为**几乎贴内容的紧凑卡片**。  
- 同高度下可多展示约 **30%～40%** 条目（48→36 行高）。  
- 面板更“轻”，左滑即决策，符合工具型弹层气质。

## 5. 方案 B：最近使用 + 全量分区（进阶）

在 A 的基础上增加一层高频区：

```
┌────────────────────────────────────────┐
│ 左滑动作                         [设置] │
│ 最近  [翻译] [Everything] [总结] [粘贴] │  ← chip，高 28，横向可折行最多 1 行
├──────────────────┬─────────────────────┤
│ Prompt           │ 工具 / Skill        │
│ …紧凑列表…       │ …紧凑列表…          │
└──────────────────┴─────────────────────┘
```

### 规则

- 记录最近 **4～6** 个动作（按 Id，持久化到 config）。  
- chip 只展示名称截断；命中区域与列表项同等可靠。  
- 列表中最近项可不重复高亮，或左侧加“·”标记，避免信息重复噪音。  
- **不把最近区做成第二套完整双栏**，高度控制在 40～48 逻辑像素。

### 收益

- 真实利用率从“可见条目数”升级为“**有效决策路径缩短**”。  
- 80% 场景 1 次滑入 + 顶部 chip 即完成。

## 6. 方案 C：非对称列宽 / 流式栅格（可选）

当一侧条目数 ≥ 另一侧 **2 倍** 时：

- **非对称**：如 Prompt 多 → 左 58% / 右 42%。  
- **或单栏流式**：全部按「分组标题 + 纵向列表」单栏展示，宽 **320～360**，更像菜单。  

手势瞄准时，单栏更窄、路径更短，但需重新校准 `ShowAtGesturePoint` 的锚点（建议仍以手势点为中心偏左上）。

**不作为首期默认**，仅在用户 Prompt/工具数量严重失衡时启用。

---

## 7. 与 AiPopup 的衔接（可选同批）

左滑选 Prompt 后进入 `AiPopup`，当前为 **上输入 43% / 下结果 57%** 固定比例。利用率问题不同（偏“结果未出前输入区过大”），建议与动作面板**分轨优化**：

| 状态 | 建议比例 |
|------|----------|
| 捕获中 / 待运行 | 输入 **70%**，结果折叠为 1 行占位 |
| 运行中 / 已有结果 | 输入 **30%～35%**，结果 **65%～70%** |
| 输入很长 | 用户可拖分栏（中期） |

首期动作面板方案**不依赖** AiPopup 改动即可独立交付。

## 8. 实现落点（Windows）

主要文件：`Quickstart/UI/AiActionPickerPopup.cs`

| 改动点 | 说明 |
|--------|------|
| `PickerLogicalSize` | 改为 `preferred` + `min/max` 计算，不再写死 520×320 |
| `EnsurePopupSizeForScreen` | 按两侧 `Controls.Count` 与行高计算高度 |
| `AddActionItem` | 行高 36/44，间距 4；预留色点绘制 |
| `ActionItemControl.OnPaint` | 标题垂直居中；Skill 才画第二行 meta |
| `CreateHeader` / `CreateToolHeader` | 压缩标题区；分组标题加计数 |
| （B）config | 增加 `RecentAiActionIds: string[]` |

macOS Avalonia 若后续做同款左滑 picker，复用同一套尺寸常量与视觉 token。

## 9. 验收标准

1. **内容 ≤4 项/侧**：面板高度明显小于 320，无明显底部空洞。  
2. **内容 ≥8 项/侧**：在 max 高度内可滚动，无需放大默认窗体。  
3. **手势命中**：松手命中成功率不低于现状（行高 ≥34～36）。  
4. **NOACTIVATE**：微信等场景仍可捕获选区。  
5. **观感**：不显挤、不错乱；高亮/圆角/类型色一致。  
6. **DPI**：`UiScaleHelper` 缩放后参数仍协调。

---

## 10. 不建议的做法

- 为“填满”而放大字号或拉高行高 → 利用率假高、真实选项变少。  
- 强行三栏 → 手势横向穿越成本高，松手误触增加。  
- 重阴影/厚边框/大插画 → 与托盘工具轻量气质冲突。  
- 默认塞进 AI 输入框预览 → 左滑第一屏变重，且捕获未完成时内容跳动。

---

## 11. 推荐实施顺序

1. **P0**：自适应高度 + 行高/间距压缩 + 分组计数 + 类型色点（方案 A）  
2. **P1**：最近使用 chip + 配置持久化（方案 B）  
3. **P2**：非对称列宽 / 单栏回退（方案 C）  
4. **P2**：`AiPopup` 运行前后动态分栏  

---

## 12. 结论

左滑面板利用率低，本质是 **固定大画布 + 松散行高 + 无内容驱动**。  
首期用 **内容自适应高度 + 36px 紧凑行 + 轻量类型色与分组计数**，即可在不大改交互模型的前提下，显著提高信息密度并保持现有圆角浅色风格。动作变多后再加 **最近使用**，把“空间利用率”升级为“决策效率”。
